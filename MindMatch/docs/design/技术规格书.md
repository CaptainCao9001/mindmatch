## 一、架构全览

### 1.1 分层架构

```
┌────────────────────────────────────────────────────────┐
│  用户交互层 (HTML Pages)                                 │
│  index.html → game1.html ~ game4.html → results.html    │
├────────────────────────────────────────────────────────┤
│  UI 渲染层 (js/ui/)                                      │
│  portrait.js  charts.js  animations.js                  │
├──────────────────┬─────────────────────────────────────┤
│  管道 A: 肖像报告  │  管道 B: 岗位匹配                    │
│  translator.js    │  integrator.js                      │
│  insight.js       │  matcher_linear.js  matcher_topsis  │
│                   │  matcher_ai.js      matcher.js      │
├──────────────────┴─────────────────────────────────────┤
│  游戏引擎层 (js/games/)                                   │
│  engine.js + game1-data.js ~ game4-data.js               │
├────────────────────────────────────────────────────────┤
│  基础设施层 (js/core/)                                    │
│  store.js    api.js    utils.js                          │
├────────────────────────────────────────────────────────┤
│  浏览器存储 (localStorage)                                │
└────────────────────────────────────────────────────────┘
```

### 1.2 数据流全链路

```
[游戏页 G1-G4]
    │  每个游戏完成时 → 写入浏览器存储
    ▼
┌──────────────┐
│ 浏览器存储     │  ← 全流程唯一数据桥梁
│ G1/G2/G3/G4  │
└──────┬───────┘
       │  results.html 加载时统一读取
       ▼
┌──────────────┐
│ integrator.js│  ← 读取4个游戏结果 → 标准化 → 14维profile
└──┬───────┬───┘
   │       │
   ▼       ▼
┌──────┐ ┌──────────────────────────┐
│管道A: │ │管道B: 岗位匹配             │
│肖像   │ │  integrator.js 输出profile│
│      │ │  → matcher_linear.js      │
│14维  │ │  → matcher_topsis.js      │
│profile│ │  → matcher_ai.js (可选)   │
│      │ │  → matcher.js 编排         │
│→translator│  → 最终排序+推荐         │
│→insight│ └──────────────────────────┘
│      │
│→portrait│
│→charts │
└──────┘
       │
       ▼
  results.html 渲染
```

### 1.3 管道 A 与管道 B 完全解耦

| | 管道 A（肖像报告） | 管道 B（岗位匹配） |
|---|---|---|
| **输入** | 同一个 14 维 profile（来自 integrator.js） |
| **处理** | translator + insight | matcher_linear + matcher_topsis + matcher_ai + matcher 编排 |
| **输出** | 肖像 HTML + 图表 | 岗位排序 + 推荐语 |
| **互依赖** | 无——A 和 B 可独立开发、独立测试、独立渲染 |
| **共享** | 仅共享输入数据（profile）和 store.js | |

#### 接口契约

```typescript
// === 公开 API ===

/** 写入一个游戏的完整结果。写入前校验数据结构。 */
function save(gameId: "game1"|"game2"|"game3"|"game4", data: GameOutput): void;

/** 读取单个游戏结果。未完成返回 null。 */
function load(gameId: string): GameOutput | null;

/** 读取全部四个游戏结果。缺失的返回 null。 */
function loadAll(): { game1: Game1Output|null; game2: Game2Output|null; game3: Game3Output|null; game4: Game4Output|null };

/** 检查四个游戏是否全部完成。 */
function hasAll(): boolean;

/** 清空所有数据（测试用）。 */
function clear(): void;

/** 获取当前已完成游戏数量。 */
function completedCount(): number;

/** 获取当前已完成的游戏 ID 列表。 */
function completedGames(): string[];
```

#### 核心算法

```
save(gameId, data):
  1. 校验 data 结构是否符合 GameOutput 类型
     校验失败 → throw Error("数据格式错误: " + msg)
  2. 写入 localStorage["mindmatch_" + gameId] = JSON.stringify(data)
  3. 更新完成列表

load(gameId):
  1. 从 localStorage 读取 "mindmatch_" + gameId
  2. 不存在 → 返回 null
  3. 存在 → JSON.parse → 返回

loadAll():
  1. 依次调用 load("game1") ~ load("game4")
  2. 返回 { game1, game2, game3, game4 }

hasAll():
  1. 检查 completedCount() === 4
```

#### 实现约束

- 存储 key 格式：`mindmatch_game1` / `mindmatch_game2` / `mindmatch_game3` / `mindmatch_game4`
- 辅助 key：`mindmatch_meta`（存完成列表和 userId）
- **生产模式**：localStorage 实现
- **测试模式**：通过构造函数注入内存 Map 实现（便于测试页使用不污染真实数据）

```javascript
// 测试模式注入
store._setBackend(memoryMap);  // 仅测试用
```

### M2: api.js — LLM 调用

| 项 | 值 |
|---|---|
| **文件** | `js/core/api.js` |
| **职责** | 封装混元 API 调用，提供超时保护、错误处理、降级机制 |
| **依赖** | 无（纯 fetch） |

#### 接口契约

```typescript
interface ApiConfig {
  hunyuanKey: string;       // 混元 API Key
  deepseekKey?: string;     // DeepSeek API Key（备用）
  hunyuanEndpoint?: string; // 混元 API 地址
  deepseekEndpoint?: string;// DeepSeek API 地址
  timeout?: number;         // 超时毫秒，默认 10000
}

/** 初始化 API 配置 */
function configure(config: ApiConfig): void;

/** 调用混元 API */
async function callHunyuan(prompt: string, options?: {
  temperature?: number;   // 默认 0.7
  maxTokens?: number;     // 默认 1024
}): Promise<string | null>;

/** 带降级的调用：混元失败自动尝试 DeepSeek */
async function callWithFallback(prompt: string, options?: {
  temperature?: number;
  maxTokens?: number;
}): Promise<string | null>;

/** 设置 API Key（从页面配置读取） */
function setApiKey(service: "hunyuan"|"deepseek", key: string): void;
```

#### 核心算法

```
callHunyuan(prompt, options):
  1. 检查 hunyuanKey 是否已配置
     未配置 → 返回 null
  2. 构建请求体：
     {
       model: "hunyuan-lite",
       messages: [{ role: "user", content: prompt }],
       temperature: options.temperature || 0.7,
       max_tokens: options.maxTokens || 1024
     }
  3. 发起 fetch(url, { method: "POST", signal: AbortController(timeout) })
  4. 超时（10s） → AbortError → 返回 null
  5. HTTP 非 200  → 返回 null
  6. 200 → 解析 response.choices[0].message.content → 返回

callWithFallback(prompt, options):
  1. result = await callHunyuan(prompt, options)
  2. result !== null → 返回 result
  3. result === null 且 deepseekKey 已配置 → await callDeepseek(prompt, options)
  4. 否则返回 null
```

#### 实现约束

- API Key 不硬编码在代码中
- 从 `localStorage["mindmatch_api_keys"]` 或页面配置面板读取
- 所有调用设置 10 秒超时
- 超时/网络错误/API 错误 → 统一返回 null（不抛异常）

```typescript
/** 范围映射：将 value 从 [inMin, inMax] 映射到 [outMin, outMax] */
function mapRange(value: number, inMin: number, inMax: number, outMin: number, outMax: number): number;

/** 归一化到 [0, 1]：value 原范围 [min, max] */
function normalize(value: number, min: number, max: number): number;

/** 分段线性映射：高分段用 */
function piecewiseHigh(score: number): number; // 0-1
/** 分段线性映射：低分段用 */
function piecewiseLow(score: number): number;  // 0-1
/** 分段线性映射：中间最优 */
function piecewiseMid(score: number): number;  // 0-1

/** 加权平均 */
function weightedMean(values: number[], weights: number[]): number;

/** 加权求和 */
function weightedSum(values: number[], weights: number[]): number;

function euclideanDistance(a: number[], b: number[]): number;

/** 简洁的 console.log 替代（上线时全局关闭） */
function log(...args: any[]): void;
```

#### 核心算法

```
normalize(value, min, max):
  返回 (value - min) / (max - min)
  夹紧到 [0, 1]

piecewiseHigh(score):  // score 范围 [0, 10]
  if score >= 7:     return 1.0
  else if score >= 4: return (score - 4) / 3
  else:               return 0.0

piecewiseLow(score):  // score 范围 [0, 10]
  if score <= 3:     return 1.0
  else if score <= 6: return (6 - score) / 3
  else:               return 0.0

piecewiseMid(score):  // score 范围 [0, 10]
  返回 1.0 - |score - 5| / 5
  夹紧到 [0, 1]
```

### M4: engine.js — 游戏引擎

| 项 | 值 |
|---|---|
| **文件** | `js/games/engine.js` |
| **职责** | 通用游戏引擎：渲染决策界面 → 计时追踪 → 按权重累计分数 → 输出维度剖面 |
| **依赖** | `js/core/utils.js`, `js/core/store.js` |

#### 接口契约

```typescript
// === 输入：游戏配置 ===
interface GameConfig {
  gameId: "game1"|"game2"|"game3"|"game4";
  title: string;                        // 游戏标题
  intro: string;                        // 引导文案（HTML）
  scenarios: Scenario[];                // 决策情境列表
  scoring: ScoringRules;                // 评分规则
  ui: {
    showTimer: boolean;                 // 是否显示计时器
    showProgress: boolean;             // 是否显示进度条
    containerId: string;               // 渲染目标 DOM ID
  }
}

interface Scenario {
  id: string;             // 如 "s1", "s2"
  title: string;          // 情境标题
  description: string;    // 情境描述（HTML）
  options: Option[];      // 可选方案
}

interface Option {
  id: string;             // 如 "s1_a", "s1_b"
  text: string;           // 选项文案
  subtext?: string;       // 选项补充说明
  weights: Record<string, number>;  // 各维度权重贡献 { "nAch": +2, "nPow": -1 }
}

interface ScoringRules {
  dimensions: string[];           // 维度 ID 列表 ["nAch","nPow","nAff"]
  rawRange: [number, number];     // 原始分数范围（用于归一化）
  outputRange: [number, number];  // 输出范围 如 [0,10] 或 [-1,1]
  // 可选：是否反转某个维度
  invert?: Record<string, boolean>;
}

// === 事件回调 ===
interface EngineCallbacks {
  onDecision?: (data: DecisionRecord) => void;
  onProgress?: (current: number, total: number) => void;
  onComplete: (result: GameOutput) => void;
}

// === 引擎入口 ===
async function run(config: GameConfig, callbacks: EngineCallbacks): Promise<void>;
```

#### 核心算法

```
run(config, callbacks):
  1. 初始化原始分数记录: rawScores = { dim1: 0, dim2: 0, ... }
  2. 渲染第一个情境到 config.ui.containerId
  3. 对每个情境 scenario:
     a. 渲染情境描述 + 选项列表
     b. 开始计时: startTime = Date.now()
     c. 监听选项点击:
        - endTime = Date.now()
        - 记录 decisionTime = endTime - startTime
        - 从 config.scoring 中查找选项权重
        - 累加到 rawScores: rawScores[dim] += option.weights[dim]
        - 记录 hoverLog（整个决策过程中每个 option 的悬停时长）
        - 触发 onDecision({ decisionId, chosenOption, decisionTime, hoverLog })
        - 触发 onProgress(current, total)
     d. 渲染下一个情境
  4. 全部完成:
     - 归一化 rawScores 到输出范围
     - 生成 behaviorSummary
     - 组装 GameOutput
     - store.save(config.gameId, GameOutput)
     - 触发 onComplete(GameOutput)
  5. 若所有四个游戏已完成 (store.hasAll() === true):
     显示"查看结果"按钮 → 跳转 results.html
```

#### 实现约束

- engine.js 不关心具体游戏内容（情境和权重全在 gameX-data.js 中定义）
- 选项权重为整数（如 +2, +1, 0, -1），引擎负责归一化
- 每个决策点的 hoverLog 记录格式: `[{ optionId: "s1_a", hoverMs: 2300 }, ...]`
- 悬停追踪：每个 option 的 mouseenter/mouseleave 事件累计
- 归一化公式: `output = (raw - minRaw) / (maxRaw - minRaw) * (outMax - outMin) + outMin`

#### game2-data.js 特殊说明

G2 的评分逻辑与 G1 不同：**放弃对应维度才扣分**（锚点的本质是"不能放弃的东西"）。

```javascript
// G2 选项的 weights 字段为负数权重
// 用户"放弃"技术路径 → TF 扣分
options: [
  { id: "s1_a", text: "放弃纯技术路线，转管理", weights: { TF: -3, GM: +2, ... } },
  { id: "s1_b", text: "坚持技术，拒绝管理诱惑",    weights: { TF: +2, GM: -2, ... } },
]
```

初始值：所有维度从满分（10）开始 → 每次"放弃"扣分 → 最终各维度独立 0-10 分。

#### game3-data.js 特殊说明

G3 不是 SJT 类型，而是**信息处理任务**。输入不是选择权重，而是**反应时 + 策略路径**。

```typescript
// G3 的每个决策点不是"选选项"，而是"完成任务"
interface CognitiveTask {
  id: string;
  type: "visual"|"semantic";  // 视觉匹配 or 语义分类
  stimulus: string;            // 刺激材料（图片 URL 或文字）
  correctAnswer: string;       // 正确答案（用于计算正确率）
  // 评分由 responseTime 和 strategy 决定
}

// G3 评分逻辑（在 engine.js 中特殊处理）:
//   responseTime_ratio = avg_global_RT / avg_detail_RT
//   wholistAnalytic = (responseTime_ratio - 1) 归一化到 [-1, +1]
//   整体型（快于全局）→ 正值；分析型（快于细节）→ 负值
```

```typescript
interface InsightInput {

interface InsightResult {

  type: "tension" | "contradiction" | "consistency" | "blindspot";
  title: string;       // 如 "内心的双重驱动"
  content: string;     // 如 "你的成就动机远高于权力动机——你追求卓越是为了..."
```

#### 核心算法

```
generate(input):
  1. 先做规则预检——是否存在触发条件：
     a. 内部张力：同游戏内两维度原始分差 ≥ 3 分
     b. 跨游戏矛盾：如 G2.AU 高但 G3.wholistAnalytic 表明分析型 → 潜在冲突
     c. 一致性确认：至少 3 个指标指向同一方向
     d. 盲点揭示：某维度 < 2 或 > 9

  2. 若无任何触发条件 → 返回 { insights: [], generated: false }
     （不浪费 API 调用）

  3. 有触发条件 → 构建 prompt：
     """
     你是心理测评解读专家。以下是一个用户的 14 维职业心理剖面和翻译结果。
     请基于以下规则生成洞察（JSON 格式）：
     - 只有在有真正发现时才输出洞察，不要硬凑
     - 每个洞察不超过 100 字
     - 洞察类型：tension / contradiction / consistency / blindspot
     - 洞察强度：strong / moderate / mild
     
     用户剖面：{JSON.stringify(profile.raw)}
     翻译结果：{JSON.stringify(translation)}
     已知触发条件：{triggers}
     """

  4. result = await api.callWithFallback(prompt)
  5. result === null → 返回 { insights: [], generated: false }（降级为纯翻译模式）
  6. JSON.parse(result) → 校验 insights 数组格式 → 返回
```


#### 接口契约

```typescript

interface LinearMatchResult {

    creative: number;       // 0-100
    analytical: number;     // 0-100
    technical: number;      // 0-100
  };

type JobType = "communication" | "creative" | "analytical" | "technical";
```

#### 核心算法

```
matchLinear(profile):
  1. 加载权重配置表（内嵌在文件中）:
     SUB_WEIGHTS = {
       communication: { nAch: 0.6, nPow: 0.7, nAff: 0.8, TF: 0.3, ... },
       creative:      { nAch: 0.7, nPow: 0.3, nAff: 0.5, TF: 0.5, ... },
       analytical:    { nAch: 0.7, nPow: 0.3, nAff: 0.4, TF: 0.8, ... },
       technical:     { nAch: 0.7, nPow: 0.3, nAff: 0.4, TF: 0.9, ... }
     }
     GAME_WEIGHTS = {
       communication: { G1: 0.30, G2: 0.25, G3: 0.20, G4: 0.25 },
       creative:      { G1: 0.30, G2: 0.25, G3: 0.25, G4: 0.20 },
       analytical:    { G1: 0.30, G2: 0.25, G3: 0.25, G4: 0.20 },
       technical:     { G1: 0.25, G2: 0.25, G3: 0.25, G4: 0.25 }
     }
     DIRECTION = {  // 岗位对每个维度的期望方向
       communication: { nAch: "high", nPow: "high", nAff: "high", ... },
       creative:      { nAch: "high", nPow: "low",  nAff: "mid",  ... },
       ...
     }

  2. 对每个岗位 jobType:
     a. 子维度匹配度计算:
        for dim in 14维:
          score = profile.dimensions[dim]  // 用户 [0,1] 值
          direction = DIRECTION[jobType][dim]
          match = mapSubDimension(score, direction)  // 分段线性映射 → [0,1]
          subScore += match * SUB_WEIGHTS[jobType][dim]

     b. 游戏级加权聚合:
        g1Score = 子维度加权平均(G1的三个维度)
        g2Score = 子维度加权平均(G2的八个维度)
        g3Score = 子维度加权平均(G3的一个维度)
        g4Score = 子维度加权平均(G4的两个维度)

     c. 最终得分:
        finalScore = g1Score*GAME_WEIGHTS[jobType].G1
                   + g2Score*GAME_WEIGHTS[jobType].G2
                   + g3Score*GAME_WEIGHTS[jobType].G3
                   + g4Score*GAME_WEIGHTS[jobType].G4
        scores[jobType] = finalScore * 100  // 转 0-100

  3. 按分数降序排序 → ranking

mapSubDimension(score, direction):
  if direction === "high":
    return piecewiseHigh(score * 10)  // 转回 0-10 走分段映射
  if direction === "low":
    return piecewiseLow(score * 10)
  if direction === "mid":
    return piecewiseMid(score * 10)
```

- 权重矩阵内嵌在文件中（不要外部加载，避免依赖）
- 权重值以 `数据结构定义.md` 第 6 节 + `四大方向特征提取.md` 第 6 节为准
- G3 的 wholistAnalytic 维度映射：分析型岗位期望高值（偏分析），创意型期望中值（偏整体）

| 项 | 值 |
|---|---|
| **文件** | `js/layers/matcher_topsis.js` |
| **职责** | 轨道二：TOPSIS 多准则决策匹配。纯函数，零依赖。 |
| **依赖** | `js/core/utils.js` |

#### 接口契约

```typescript

// UnifiedProfile.dimensions + ideal-profiles.json 中的理想剖面

// === 输出 ===
interface TopsisResult {
  scores: {
    communication: { closeness: number; dPlus: number; dMinus: number };
    creative:      { closeness: number; dPlus: number; dMinus: number };
    analytical:    { closeness: number; dPlus: number; dMinus: number };
    technical:     { closeness: number; dPlus: number; dMinus: number };
  };
  ranking: JobType[];  // 按 closeness 降序
}

```

#### 核心算法

```
matchTopsis(profile):
  // 预置：4 个岗位的 14 维理想剖面
  IDEAL = {
    communication: [0.6, 0.7, 0.8, 0.3, 0.7, 0.4, 0.5, 0.6, 0.8, 0.5, 0.5, 0.5, 0.7, 0.5],
    creative:      [0.7, 0.3, 0.5, 0.5, 0.4, 0.8, 0.3, 0.9, 0.4, 0.7, 0.6, 0.3, 0.6, 0.8],
    analytical:    [0.7, 0.3, 0.4, 0.8, 0.4, 0.5, 0.6, 0.3, 0.3, 0.6, 0.4, 0.8, 0.5, 0.4],
    technical:     [0.7, 0.3, 0.4, 0.9, 0.5, 0.5, 0.5, 0.5, 0.3, 0.6, 0.4, 0.7, 0.5, 0.5]
  }
  // 14 维顺序: nAch, nPow, nAff, TF, GM, AU, SE, EC, SV, CH, LS, W-A, presence, search
  DIMENSION_WEIGHTS = [0.07, 0.05, 0.08, 0.08, 0.07, 0.07, 0.05, 0.07, 0.08, 0.06, 0.05, 0.10, 0.08, 0.09]

  userVector = 用户 14 维向量 [dimensions.nAch, dimensions.nPow, ..., dimensions.search]

  // Step 1: 构建决策矩阵 D[4×14]
  for i in 0..3:
    for j in 0..13:
      D[i][j] = 1 - abs(userVector[j] - IDEAL[jobTypes[i]][j])
      // 匹配度 = 距离理想值的反向距离

  // Step 2: 向量归一化
  for j in 0..13:
    colSum = sqrt(sum(D[i][j]^2 for i in 0..3))
    for i in 0..3:
      R[i][j] = D[i][j] / colSum

  // Step 3: 加权
  for i in 0..3:
    V[i][j] = R[i][j] * DIMENSION_WEIGHTS[j]

  // Step 4: 理想解
  Aplus[j]  = max(V[i][j] for i in 0..3)
  Aminus[j] = min(V[i][j] for i in 0..3)

  // Step 5-6: 距离 + 贴近度
  for i in 0..3:
    dPlus[i]  = sqrt(sum((V[i][j] - Aplus[j])^2))
    dMinus[i] = sqrt(sum((V[i][j] - Aminus[j])^2))
    closeness[i] = dMinus[i] / (dPlus[i] + dMinus[i])

  // Step 7: 排序
  ranking = jobTypes 按 closeness 降序排列
```

#### 实现约束

- 理想剖面值从 `匹配算法设计.md` 第 3.2 节取值，内嵌在代码中
- 维度全局权重从 `匹配算法设计.md` 推导，内嵌在代码中
- closeness 范围 [0,1]，越接近 1 越匹配

```
     discrepancy = 是否第一名的岗位不同 OR 前两名顺序颠倒
     
  2. 若 discrepancy === false:
     return { triggered: false }  // 两种方法一致，不需要 AI

  3. 若 topsisResult 第一名 closeness < 0.6:
     自动触发 AI（得分过低，不确定性高）

  4. 构建 prompt：
     """
     你是职业匹配评审专家。
     用户的 14 维心理剖面：
     {JSON.stringify(input.profile.dimensions)}
     
     线性加权结果：{input.linearResult.scores}
     TOPSIS 结果：{input.topsisResult.scores}
     
     两种方法排名不一致。请评估哪个结果更合理，并给出每个岗位的置信度。
     
     输出 JSON：
     {
       "confidence": { "communication": 0.9, "creative": 0.6, ... },
       "reasoning": "...",
       "flags": ["需要注意的标记"]
     }
     """

  5. result = await api.callWithFallback(prompt)
  6. result === null → return { triggered: true, ... } (AI 失败，标记但不影响)
  7. JSON.parse → 返回
```

#### 降级策略

```
AI 不触发 → triggered=false → 结果页直接展示 TOPSIS 结果
AI 失败   → triggered=true 但没有 confidence → 结果页展示"匹配结果仅基于算法分析"
AI 成功   → 附加 confidence + reasoning + flags → 结果页展示完整匹配报告
```

### M20: matcher.js — 匹配编排

| 项 | 值 |
|---|---|
| **文件** | `js/layers/matcher.js` |
| **职责** | 主编排器：调用三个轨道 → 交叉验证 → 输出最终匹配结果。 |
| **依赖** | `js/layers/matcher_linear.js`, `js/layers/matcher_topsis.js`, `js/layers/matcher_ai.js` |

#### 接口契约

```typescript
// === 输入 ===
// UnifiedProfile

// === 输出 ===
interface MatchResult {
  userId: string;
  generatedAt: number;
  primary: JobType;                    // 最优匹配岗位
  ranking: JobType[];                  // 四个岗位排序
  details: Record<JobType, MatchDetail>;
  aiAssisted: boolean;                 // 是否有 AI 参与
  aiNote?: string;                     // AI 补充说明
}

interface MatchDetail {
  linearScore: number;       // 线性得分 0-100
  topsisCloseness: number;   // TOPSIS 贴近度 0-1
  aiConfidence?: number;     // AI 置信度 0-1（可选）
  reason: string;            // 匹配理由（规则生成的基础理由）
  flags: string[];           // 注意事项
}

async function match(profile: UnifiedProfile): Promise<MatchResult>;
```

#### 核心算法

```
async match(profile):
  1. 轨道一（同步）:
     linearResult = matchLinear(profile)
  
  2. 轨道二（同步）:
     topsisResult = matchTopsis(profile)
  
  3. 轨道三（异步，可选）:
     aiResult = await matchAi({ profile, linearResult, topsisResult })
  
  4. 交叉验证：
     linearTop = linearResult.ranking[0]
     topsisTop = topsisResult.ranking[0]
     
     if linearTop === topsisTop:
       primary = linearTop  // 高可信
       ranking = topsisResult.ranking  // 用 TOPSIS 排序（更精细）
     else:
       // 不一致：优先 TOPSIS（学术验证更充分），标记为需要 AI
       primary = topsisTop
       ranking = topsisResult.ranking
       // 如 AI 有结果，用 AI 置信度加权修正
  
  5. 组装理由:
     for each jobType:
       reason = 规则生成基础理由（模板: "{最高贡献维度}得分较高，符合{jobType}型核心特质"）
       若有 AI → 附加 aiReasoning
  
  6. 返回 MatchResult

基础理由模板:
  communication: "亲和动机({nAff})和服务锚点({SV})得分突出，适合以人际连接为核心的沟通型岗位"
  creative:      "创业锚点({EC})和寻求意义({search})得分较高，适合以创造新事物为驱动的创意型岗位"
  analytical:    "成就动机({nAch})和技术锚点({TF})得分较高，认知风格偏分析型，适合以数据洞察为核心的分析型岗位"
  technical:     "技术锚点({TF})和创业锚点({EC})得分较高，适合以技术实现为驱动的技术应用型岗位"
```

#### 实现约束

- 轨道一、二是同步纯函数，轨道三是异步可选的
- `match()` 为 async 函数（等轨道三的 promise）
- 即使轨道三失败，`match()` 也返回有效结果（aiAssisted=false）
- 所有结果写回 store（`store.save("match", MatchResult)`），供 results.html 读取

### 3.1 全流程数据可用性

| 阶段 | 数据存储位置 | 谁能读取 | 何时写入 |
|------|------------|---------|---------|
| G1 完成 | localStorage["mindmatch_game1"] | store.load("game1") | 引擎 onComplete 回调 |

| G4 完成 | localStorage["mindmatch_game4"] | store.load("game4") | 引擎 onComplete 回调 |
| 结果页加载 | — | store.loadAll() | — |
| 整合完成 | 内存 UnifiedProfile | 所有后续模块 | integrator.integrate() |
| 翻译完成 | 内存 TranslationResult | portrait.js, insight.js | translator.translate() |
| 匹配完成 | localStorage["mindmatch_match"] | results.html | matcher.match() |

### 3.2 数据完整性检查点

```
1. 每个游戏完成时:
   store.save() 前 → 校验结构是否符合 GameOutput 类型

2. results.html 加载时:
   integrator.integrate() → 始终返回 UnifiedProfile（支持部分完成）
   已完成维度正常显示，未完成维度显示占位提示

3. integrator.integrate() 执行时:
   已完成的游戏 → 标准化到 dimensions 和 raw
   未完成的游戏 → 对应维度设为 null

4. match 执行时:
   若 profile.meta.allCompleted !== true → 不执行匹配，显示占位

5. insight 执行时:
   api.callWithFallback 返回 null → 降级，不崩溃
```

### 3.3 数据一致性约定

- **不可变性**：所有 `js/layers/` 中的函数接收输入 → 返回新对象，不修改输入
- **单一真相源**：浏览器存储是游戏层数据的唯一写入者，其他模块只读
- **幂等性**：`store.save()` 可多次调用，后一次覆盖前一次
- **版本标记**：每个 GameOutput 包含 `version: "1.0"` 字段，防止跨版本兼容问题

### 4.1 关键连接点

```
【连接点 1】游戏 → 浏览器存储

  注意: 始终返回，未完成维度为 null

  类型: UnifiedProfile

【连接点 6】管道 A 输出 → UI
  类型: TranslationResult + InsightResult
  UI接收: portrait.renderPortrait(containerId, data)

【连接点 7】管道 B 输出 → UI
  类型: MatchResult
  UI接收: charts.renderMatchChart(containerId, matchResult)
```
