> 评委评分维度：思辨深度、创意巧思、功能完整度、交互体验、落地可行性

## 第一部分：项目立意与需求分析

### 1.1 核心问题

**简历只能告诉雇主你过去做了什么，但它无法回答更本质的问题：**

- 这个人真正在乎什么？（动机与价值观）
- 这个人习惯怎么思考？（认知风格）
- 这个人在什么环境中会 thriving？（适配性）
- 这个人自己知道自己适合什么吗？（自我觉察）

大量求职者找不到真正适合自己的工作，不是因为能力不够，而是因为**他们不了解自己的深层特质，也不知道这些特质和什么岗位最匹配**。

### 1.2 核心定位

**MindMatch 不是一个招聘工具，而是一个自我觉察工具。**

> "帮你看到自己不曾察觉的地方，让你更准确地找到适合的方向。"

差异化本质：
- pymetrics 的叙事："公司用科学方法评估你" → 用户心态：被审视
- MBTI 的叙事："你是 XX 类型的人" → 用户心态：被分类
- MindMatch 的叙事："我们在帮你理解自己" → 用户心态：被理解

### 1.3 产品形态

用户通过 4 个投射式小游戏（总时长 10-15 分钟），在沉浸式体验中自然暴露自己的深层特质。系统采集行为数据（选择内容 + 鼠标停留时长 + 决策时长），提供实时洞察反馈，最终通过 **AI 生成个性化自我觉察报告 + 岗位匹配分析**。

### 1.4 评估维度（理论基础）

基于心理学理论整合，确定 4 个核心评估维度：

| # | 维度名称 | 回答的问题 | 理论来源 |
|---|---------|-----------|---------|
| ① | 核心驱动力 | "你为什么而做？什么在真正驱动你？" | McClelland 需求动机理论（成就/亲和/权力）、London 职业动机理论 |
| ② | 不可放弃的锚点 | "你绝对不会妥协的是什么？" | Schein 职业锚理论（MIT 纵向追踪研究） |
| ③ | 认知处理风格 | "你习惯怎么理解世界？" | Witkin 认知风格理论（场独立/场依存）、Riding 整体型/分析型 |
| ④ | 意义建构取向 | "你在工作中寻找什么意义？" | Schneider P-O Fit 价值观匹配、意义建构理论 |

### 1.5 匹配目标——腾讯 AI-HR 四象限

| 岗位类型 | Slogan | 核心驱动力 | 锚点 | 认知风格 | 意义取向 |
|---------|--------|-----------|------|---------|---------|
| 沟通型 | 用 AI 助力人与业务的链接 | 亲和动机（连接人） | 服务/奉献 + 生活平衡 | 场依存 + 整体型 | 影响他人 |
| 创意型 | 用 AIGC 释放创意 | 自主/内在驱动（创造） | 自由/独立 + 创业创新 | 整体型 + 高灵活性 | 创造新事物 |
| 分析型 | 用 AI 洞悉数据真相 | 成就动机（揭示真相） | 专业深度 + 纯粹挑战 | 场独立 + 分析 + 反思 | 揭示规律 |
| 技术应用型 | 用技术重塑管理 | 胜任动机（做出东西） | 技术精通 + 应用创新 | 分析型 + 迭代式 | 解决实际问题 |

> 详见：`四大方向特征提取.md`

### 1.6 用户旅程

```
打开页面 → 简短引导（"这不是测试，是帮你了解自己"）
    → 游戏1：核心驱动力（3-4分钟）
    → 游戏1结尾：行为触发式洞察反馈
    → 游戏2：不可放弃的锚点（3-4分钟）
    → 游戏2结尾：行为触发式洞察反馈
    → 游戏3：认知处理风格（3分钟）
    → 游戏3结尾：行为触发式洞察反馈
    → 游戏4：意义建构取向（3分钟）
    → 游戏4结尾：行为触发式洞察反馈
    → 生成报告（AI 驱动）：思维肖像 + 岗位匹配分析
```

### 1.7 已砍掉的功能

- ❌ 简历上传与解析（开发成本高，对评委评分权重低）
- ❌ 复杂的 ML 模型（demo 阶段用规则引擎 + AI 即可）
- ❌ 复杂实时反馈（只做行为触发式简单反馈，不做实时 AI 对话）

## 第二部分：技术架构

### 2.1 技术栈

| 层级 | 技术选型 | 选型理由 |
|------|---------|---------|
| **前端框架** | 纯 HTML + CSS + JavaScript（轻量级） | 14天单人选不出错；无构建工具依赖；部署简单 |
| **图表可视化** | ECharts（CDN 引入） | 雷达图开箱即用；中文文档完善；体积可控 |
| **行为数据采集** | 自研轻量 SDK | 记录鼠标停留时长、决策时长、选择内容；存入 localStorage |
| **评分引擎** | 规则引擎（if-else 加权） | 逻辑透明可解释；评委能看懂；demo 阶段不需要 ML |
| **AI 报告生成** | 腾讯混元（主）/ DeepSeek V3（备） | 见 2.2 详细说明 |
| **部署** | Vercel（免费额度） | 静态网站一键部署；自动 HTTPS；公网可访问 |

### 2.2 AI 选型详细说明

**方案：规则评分 + AI 报告生成**

系统分为两层：
1. **评分层（规则引擎）**：用户行为数据 → 4 个维度分值 → 4 类岗位匹配度。逻辑透明、可解释、可审计。
2. **表达层（LLM）**：将结构化评分转化为个性化的自我觉察报告文案。AI 擅长生成自然语言洞察，提升"被说中感"。

**API 选型：**

| 优先级 | API | 价格 | 选型理由 |
|-------|-----|------|---------|
| **主选** | 腾讯混元大模型 | 新用户免费额度，之后按量计费 | 与项目方（腾讯 AI-HR）技术生态对齐；方案说明中可明确写出品牌对齐理由 |
| **备选** | DeepSeek V3 API | 约 ¥1/百万 tokens | 兼容 OpenAI 格式，切换成本低；中文质量顶尖；几乎等于免费 |

**接入方式：**
- 两者均兼容 OpenAI API 格式，代码中封装统一的 `generateReport(data)` 函数
- 配置 fallback 机制：混元调用失败自动切换 DeepSeek
- API Key 通过环境变量注入，不硬编码
- 每个用户只在最后一步调用一次 AI（4 个维度评分 + 匹配结果 → ~1000 tokens 输入 → ~800 tokens 输出）

### 2.3 数据流架构

```
用户行为数据（选择 + 时长 + 鼠标停留）
    ↓ localStorage 存储各游戏数据
规则引擎评分
    ↓ 4 个维度分值（0-100）
加权匹配算法
    ↓ 4 类岗位匹配度（0-100）
AI 报告生成
    ↓ 个性化自我觉察文案 + 岗位匹配分析
报告页面渲染
    ↓ ECharts 雷达图 + 文字排版
```

### 2.4 部署方案

| 步骤 | 操作 |
|------|------|
| 1 | GitHub 创建仓库，推送代码 |
| 2 | Vercel 连接 GitHub 仓库，自动部署 |
| 3 | 环境变量配置：混元 API Key、DeepSeek API Key |
| 4 | 自定义域名（可选）或使用 Vercel 免费域名 |
| 5 | 验证公网可访问性 |

**注意：AI API 调用在用户浏览器端执行（前端直接调用），不经过服务器。** Demo 阶段无后端，API Key 暴露风险可接受（提交说明中注明）。

## 第七部分：评分策略建议

> 核心思路：**两科满分 + 三科及格**，不追求全面高分。

| 评委维度 | 策略 | 如何拿分 |
|---------|------|---------|
| **思辨深度** | 🎯 主攻 | 方案说明中写透理论框架；自我觉察理念讲清楚；AI 报告中有深度洞察 |
| **创意巧思** | 🎯 主攻 | "帮用户理解自己"的定位差异；投射式游戏设计；行为触发反馈 |
| **功能完整度** | 🎯 保及格 | 4 个游戏能跑通 + AI 报告能生成 + 数据流完整 + 公网可访问 |
| **交互体验** | 🎯 保及格 | 流程顺畅不卡顿；反馈文案到位；基本美观 |
| **落地可行性** | 🎯 保及格 | 规则评分逻辑透明；AI 选型有理有据；方案书中标注"假设模型需数据验证" |

**AI 工具选型理由（方案说明核心卖点之一）：**

> 本项目采用"规则引擎 + LLM"双轨架构。评分层使用透明可解释的规则引擎，确保匹配逻辑可审计、可迭代；表达层使用腾讯混元大模型，将结构化评分转化为个性化的自我觉察文案。选型考虑——混元大模型与项目方技术生态对齐，且在中文自然语言生成上有良好表现。备选 DeepSeek V3 作为 fallback，两者均兼容 OpenAI API 格式，切换成本极低。
