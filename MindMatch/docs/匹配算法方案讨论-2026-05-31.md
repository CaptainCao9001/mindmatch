# MindMatch 职业匹配算法方案讨论

> 状态：讨论文档，决策后执行
> 日期：2026-05-31
> 议题：如何用 14 维心理测量结果匹配 6 个模糊方向 + 54 个具体职业

---

## 一、现状数据结构

```
用户做完 4 个游戏
  → integrator.js 输出 14 维 UnifiedProfile（每维 [0,1]）
  → matcher.js 双轨引擎：
      轨道一：线性加权（mapSubDimension → 加权求和 → 0-100）
      轨道二：TOPSIS（匹配矩阵 → 向量归一化 → 加权 → 欧氏距离 → 贴近度）
  → 融合：TOPSIS 定排名，线性定分数，AI 审核触发检测
```

### 当前匹配函数 `mapSubDimension(actual, direction)`

```js
direction = 'high'  → actual ≥ 0.7 满分, 0.4-0.7 线性, <0.4 零分
direction = 'low'   → actual ≤ 0.3 满分, 0.3-0.6 线性, >0.6 零分
direction = 'mid'   → 接近 0.5 最好, 用三角分布
```

这是**分区匹配（zone-based）**——不看精确的 ideal 值，只看"用户在不在对的区间里"。

## 二、核心设计问题

### Q1：两级匹配 vs 扁平匹配

| | 方案 A：扁平匹配 | 方案 B：两级匹配（推荐） |
|---|---|---|
| **怎么做** | 54 个候选（6方向×9职业）直接 TOPSIS 排名 | 第一级：6 方向排名 → 第二级：每方向内 9 职业排名 |
| **候选数** | 54 | 6（方向）+ 9×3（Top 3 方向） |
| **TOPSIS 质量** | 54 候选时正理想解是"合成体"，不真实 | 6 或 9 候选时正理想解是真实存在的剖面 |
| **用户体验** | 看到一堆职业列表，没有"方向归属感" | "你是创意塑造者(82%)"+"具体职位: UI设计师(85%)" |
| **心理学含义** | 方向只是分组标签 | 方向是原型，职业是原型的具体化 |
| **代码改动** | 小（只改数据文件） | 中（需两层编排） |

**推荐：方案 B（两级匹配）**

理由：方向是整个产品概念的锚点——"模糊方向"是一个心理学原型，用户先认同方向再探索具体职位，比直接看职业列表更有共鸣。

### Q2：匹配函数——保持分区匹配还是改为距离匹配？

| | A：分区匹配（当前） | B：距离匹配 | C：模糊隶属度 |
|---|---|---|---|
| **公式** | `fit(实际值, 方向) =` 分段线性 | `fit = 1 - \|实际值 - 理想值\|` | 梯形隶属度函数 |
| **精度** | 粗（三档） | 细（连续） | 中等 |
| **鲁棒性** | ✅ 对测量误差不敏感 | ⚠️ 对数值偏差敏感 | ✅ 有过渡带 |
| **解释性** | ✅ "你在这个维度倾向高值" | ✅ "你与理想值差0.15" | ❌ 用户不懂隶属度 |
| **实现** | 已有，零改动 | 需新增 | 需新增 |

**推荐：分区匹配（A）保持不变**

理由：
1. 当前代码已实现且验证通过
2. 心理测量分数本身不精确，分区比精确距离更稳健
3. 6 个方向的心理剖面在设计上已经足够区分——不需要精确到小数点来拉开差距

### Q3：方向 vs 具体职业的关系怎么建模？

**原型理论视角**：方向是"原型"(prototype)，具体职业是原型的"实例化"(instantiation)。

```
创意塑造者（原型）
├─ 基线 14 维剖面                      ← 定义"创意塑造者是什么"
├─ UI 设计师     TF↑0.15  wholist↑0.10  ← 比原型多系统性+技术感
├─ 内容创作者   AU↑0.10  wholist↓0.10  ← 比原型更自主+更整体思维
├─ 产品经理     GM↑0.15  SE↑0.10       ← 比原型更稳定+管理感
├─ 编导/视频    EC↑0.10  presence↑0.05
├─ 品牌策划     nPow↑0.10  SV↑0.05
├─ UX 设计师    TF↑0.05  nAff↑0.05
├─ 插画师       AU↑0.10  TF↓0.05
├─ 广告创意总监 nPow↑0.15  GM↑0.10
└─ 自媒体运营   EC↑0.15  SE↓0.05

所有职业都继承方向的 direction（high/mid/low），用 ±delta 微调 ideal 值
```

**这样做的好处**：
- 只需定义 6 个完整剖面 + 54 个小 delta（而非 54 个完整剖面）
- 新增职业只需在方向基础上加减几个维度的偏移
- 维护成本低：如果调整方向剖面，所有职业自动继承
- 心理学上有依据：原型理论在认知心理学中广泛验证

### Q4：引擎架构怎么改？

#### 当前架构

```
matcher.js
├─ matcher_linear.js  → 线性加权（14维 → 4岗位）
└─ matcher_topsis.js  → TOPSIS（14维 → 4岗位）
```

#### 改为两级架构

```
matcher.js                    ← 主编排器（保持不变）
├─ matcher_direction.js       ← [新增] 14维 → 6方向排名
│   └─ 内部复用 TOPSIS + 线性加权双轨
│       candidates = 6 个方向剖面
│       输出：方向排名 + 匹配度
│
└─ matcher_jobs.js            ← [新增] 14维 → 指定方向内的具体职业排名
    └─ 内部复用 TOPSIS + 线性加权双轨
        candidates = 该方向下的 n 个职业剖面
        输出：职业排名 + 匹配度
```

**或者，更简洁的方式**——当前的 `matcher_topsis.js` 和 `matcher_linear.js` 本身是通用的（接收任意数量的 candidates），不耦合 4 个岗位。所以可以：

```js
// 不新增文件，只在 matcher.js 编排层扩展
match(profile, directions, directionJobs) {
  // Level 1: 所有方向排名
  const directionRanking = topsisMatch(profile, directions);
  
  // Level 2: Top 3 方向的具体职业排名
  const jobRankings = {};
  for (const dir of directionRanking.slice(0, 3)) {
    jobRankings[dir.jobId] = topsisMatch(profile, directionJobs[dir.jobId]);
  }
  
  return { directionRanking, jobRankings };
}
```

**改动范围**：
- `matcher.js`：扩展主编排逻辑（~30 行）
- `matcher_linear.js` / `matcher_topsis.js`：**不改**
- `ideal-profiles.json`：新增 `directions` + `directionJobs` 结构
- `results.html` / `match-ui.js`：扩展渲染逻辑

### Q5：权重策略

当前系统有两层权重：
1. `profiles[].weight` — 每维在岗位内部的权重（"TF 对这个岗位多重要"）
2. `gameWeights` — 每游戏在岗位中的权重（"G3 对这个岗位多诊断性"）

对于 6 个方向：

**per-dimension weight 原则**：区分度越高的维度权重越大
- 使方向之间拉开距离

```
系统建构者：TF(0.30) + wholistAnalytic(1.00) ← 最高区分度维度
深度解读者：nAch(0.40) + wholistAnalytic(1.00)
创意塑造者：AU(0.30) + EC(0.28) + search(0.55)
人际联结者：nAff(0.35) + SV(0.30) + presence(0.55)
价值驱动者：nPow(0.40) + GM(0.30) + presence(0.50)
赋能陪伴者：nAff(0.40) + SV(0.35) + presence(0.60)
```

**gameWeights 原则**：该游戏对该方向的诊断性

```
                    G1(驱动力)  G2(职业锚)  G3(认知)  G4(意义)
系统建构者          0.20        0.20        0.35      0.25
深度解读者          0.25        0.20        0.30      0.25
创意塑造者          0.25        0.25        0.20      0.30
人际联结者          0.30        0.30        0.15      0.25
价值驱动者          0.30        0.30        0.15      0.25
赋能陪伴者          0.25        0.25        0.20      0.30
```

## 三、算法选择对比矩阵

| 维度 | 分区匹配 + TOPSIS | 距离匹配 + TOPSIS | 模糊隶属度 + TOPSIS | 纯线性加权 |
|---|---|---|---|---|
| 对测量误差的鲁棒性 | ⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐ | ⭐⭐ |
| 方向区分度 | ⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ |
| 用户可解释性 | ⭐⭐⭐ | ⭐⭐ | ⭐ | ⭐⭐⭐ |
| 实现复杂度 | 已有 | 低 | 中 | 已有 |
| 与现有代码兼容 | 100% | 需改 mapSubDimension | 需重写 | 100% |
| 学术界依据 | TOPSIS+MCDM | O*NET 距离法 | 模糊多准则决策 | SAW 法 |

**结论：保持当前分区匹配 + TOPSIS 双轨算法，扩展到两级结构。**
