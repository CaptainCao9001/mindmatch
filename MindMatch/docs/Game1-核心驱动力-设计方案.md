# Game 1：核心驱动力 — 设计方案

> 创建日期：2026-05-30
> 状态：方案设计完成，待实施
> 对应技术规格书：M4（engine.js）、M5（game1-data.js）、M9（game1-core-drive.html）

---

## 一、设计定位

| 属性 | 值 |
|------|-----|
| **学术框架** | McClelland 三动机理论（成就动机 nAch / 权力动机 nPow / 亲和动机 nAff） |
| **游戏类型** | 情境判断测验（SJT）— 角色代入 + 两难选择 |
| **输出** | 3 个维度 × 0-10 连续分数 + 行为摘要 |
| **目标时长** | 3-5 分钟（5 个决策点 × 30-60 秒/个） |
| **核心叙事** | 你是一个新晋项目负责人，带领 5 人小团队，面临一系列真实管理困境 |

---

## 二、整体交互流程

```
入场引导 → 情境 1 → 反馈 1 → 情境 2 → 反馈 2 → ... → 情境 5 → 反馈 5 → 过渡动画 → 结果摘要 → 返回首页
```

这是「渐进式叙事」结构 —— 不是冷冰冰的问卷，而是 5 个连续的职场情境，让用户沉浸在一个故事弧线中。

---

## 三、5 个情境设计（叙事弧线）

| # | 情境标题 | 叙事推进 | 决策压力 | 核心测量 | 选项数 |
|---|---------|---------|---------|---------|:-----:|
| S1 | 加班还是放手 | 上任第一天：前任留下一个烂摊子，deadline 三天后 | 中等 | nAch vs nAff | 4 |
| S2 | 带人还是自己干 | 团队里有能力参差不齐的成员，核心任务需要高质量交付 | 中等 | nAch vs nPow | 4 |
| S3 | 冲突还是和谐 | 两位核心成员因为技术方案激烈争吵，项目停滞 | 高 | nAff vs nPow | 4 |
| S4 | 升职还是带团队 | 领导私下暗示你申请管理岗，但你很喜欢做技术 | 高 | nAch vs nPow | 3 |
| S5 | 庆功还是复盘 | 项目成功交付，团队提议聚餐庆祝，但你觉得还有很多可改进的 | 低（尾声） | nAch vs nAff | 3 |

### 设计原则（基于认知心理学 + 游戏化设计）

1. **叙事递进**：压力从低到高再到低，符合 Peak-End Rule —— 高潮在 S3/S4，结尾 S5 轻松收尾，留下积极印象
2. **选项无对错**：每个选项都有合理理由，不存在"标准答案"（避免社会赞许性偏差）
3. **隐式测量**：用户不知道自己正在被测量什么动机，避免策略性回答
4. **维度混合**：5 个情境不是机械地"每个维度各测一次"，而是混合测量，通过后端的权重累积来分离维度分数

### 4.2 选项交互设计

| 设计决策 | 选择 | 理由 |
|---------|------|------|
| 选项展示方式 | **卡片式**（非单选按钮） | 认知心理学：卡片比 radio button 更有"选择感"，降低"我在做测试"的心理暗示 |
| 点击后行为 | **选中高亮 + 1.5s 短暂停留后自动进入下一题** | Hick's Law：选中即进入，减少二次确认的认知负担；短暂停留让用户看到自己的选择 |
| 选项顺序 | **每次随机打乱** | 避免首因效应（primacy effect），确保权重分布不受位置影响 |
| 鼠标悬停追踪 | **静默记录**（用户无感知） | hoverLog 仅供信度验证，UI 上不显示任何追踪提示 |
| 决策耗时显示 | **不显示计时器** | 显示计时器会引发焦虑，导致策略性快速回答 |
| 进度指示 | **顶部 5 个圆点** | 比 progress bar 更轻量，与首页的进度概览风格一致 |

### 4.3 选项权重设计（S1 示例）

```javascript
// S1: 加班还是放手
{
  id: "s1",
  options: [
    { id: "s1_a", text: "通宵三天，亲自把核心模块重写", subtext: "质量是我的底线", weights: { nAch: +3, nPow: 0, nAff: -1 } },
    { id: "s1_b", text: "召集团队，重新分工，我来统筹全局", subtext: "让对的人做对的事", weights: { nAch: +1, nPow: +2, nAff: +1 } },
    { id: "s1_c", text: "先找每个人单独聊，了解他们的想法", subtext: "人心稳了项目才能稳", weights: { nAch: 0, nPow: 0, nAff: +3 } },
    { id: "s1_d", text: "向领导如实说明情况，争取延期", subtext: "实事求是比硬撑更重要", weights: { nAch: -1, nPow: +1, nAff: +1 } },
  ]
}
```

**权重设计原则**：
- 每个选项对 3 个维度都有权重贡献（含 0 和负数）
- 每个维度在每个情境中的 4 个选项权重和 ≈ 0（保持平衡）
- 避免极端权重（最大 +3，最小 -1），防止单次选择过度影响总分

## 五、行为触发式洞察反馈

每个决策后，显示一段简短的非评判性反馈。

### 5.1 反馈设计原则

| 原则 | 实践 |
|------|------|
| **不揭示分数** | 反馈不说"你的成就动机 +3"，而是描述行为模式 |
| **不评判好坏** | 不说"这个选择很好/不好"，而是说"你选择了 X" |
| **引发好奇** | 用陈述句制造"被说中感"，激发用户继续玩下去的好奇心 |
| **简短克制** | 2-3 句话，不超过 3 秒阅读时间 |
| **可选跳过** | 用户可以直接点击"继续"，反馈不会阻塞流程 |

### 5.2 反馈示例

| 用户选择 | 反馈文案 |
|---------|---------|
| S1-A（通宵重写） | "你选择了扛下来。对质量有执念的人，往往在看不见的地方也对自己有着最高的要求。" |
| S1-B（重新分工统筹） | "你第一反应是看清全局再行动。这说明你习惯站在'掌控者'的位置思考问题。" |
| S1-C（先聊天安抚） | "你注意到人的状态优先于任务的进度。在你看来，关系是一切的基础。" |
| S1-D（如实汇报） | "你选择了透明的沟通。比起自己扛，你更相信'说清楚'本身就是一种力量。" |

## 六、入场引导页设计

用户从首页点击 G1 卡片后，首先看到一个过渡引导页：

```
┌─────────────────────────────────────────────┐
│                                              │
│         ── Game 1 / 4 ──                     │
│                                              │
│         核心驱动力                            │
│                                              │
│    你将扮演一个新晋项目负责人。                │
│    在接下来的 5 个场景中，                    │
│    你需要做出真实的选择。                      │
│                                              │
│    没有正确答案。                             │
│    选你第一反应想做的那个。                    │
│                                              │
│         ⏱ 预计 3–5 分钟                      │
│                                              │
│         [准备好了，开始]                       │
│                                              │
└─────────────────────────────────────────────┘
```

**设计要点**：
- 入场动画：淡入 + 轻微上移（与首页风格一致）
- "没有正确答案" —— 关键的心理学暗示，降低测试焦虑
- "选你第一反应" —— 鼓励直觉选择，减少策略性思考
- 简短：整个页面不超过 6 行文字（Cognitive Load Theory：减少外在负荷）

## 十、与参考项目的对标分析

| 设计维度 | MindMatch G1 | Test Partnership MindmetriQ | 16Personalities | practiceaptitudetests |
|---------|:-----------:|:-------------------------:|:-------------:|:-------------------:|
| 交互方式 | 卡片选择 | 拖放/滑动/旋转 | Likert 7 点量表 | 单选 + 排序 |
| 场景呈现 | 叙事化情境（连续故事） | 独立小游戏 | 陈述句 | 文本段落 + 配图 |
| 进度显示 | 圆点指示器 | 每个小游戏独立进度 | 进度条 | 数字步进 |
| 计时器 | 无（静默记录） | 有（短时间限制） | 无 | 无 |
| 决策后反馈 | 行为触发式洞察 | 得分/无反馈 | 无 | 详细解析 |
| 氛围 | 温暖/叙事/沉浸 | 游戏化/挑战 | 专业/简洁 | 教育/练习 |
| 预计时长 | 3-5 分钟 | 4-6 分钟 | 10 分钟 | 20+ 分钟 |

**MindMatch 的差异化**：
- **叙事连续性**：5 个情境构成一个完整故事弧线（同类产品都是独立题目）
- **行为触发式反馈**：每个决策后给一段洞察文案（同类产品要么无反馈，要么是教育性解析）
- **零焦虑设计**：无计时器、无正确答案、不显示分数（同类产品多多少少有考试感）

## 十一、认知心理学审查

### ✅ 做对了的

1. **工作记忆保护**：每次只展示 1 个情境 + 4 个选项（Cowan 限制：4 chunks），远低于认知过载阈值
2. **选择架构合理**：4 个选项在 Hick's Law 的"甜蜜区"——多于 3 个有真实选择感，少于 7 个无决策瘫痪
3. **认知负荷控制**：引导页 ≤6 行文字，反馈 ≤3 句，每个决策 ≤2 屏信息量
4. **Peak-End Rule**：高潮在 S3/S4（高压力），结尾 S5 轻松 + confetti 动画，整体体验会被记忆为积极
5. **零暗模式**：无排行榜、无惩罚机制、无强制连续性，用户随时可以退出

### ⚠️ 需要注意的

1. **选项打乱可能影响可读性**：随机排列后，选项的逻辑顺序可能被打乱，需要在 shuffle 后检查是否存在明显的"最佳位置"效应
2. **反馈文案需要精心打磨**：2-3 句话要在"被说中感"和"不评判"之间找平衡，这需要多轮人工测试
3. **叙事沉浸 vs 测量效度的张力**：故事越沉浸，用户越可能"进入角色"而非"表达自我"，需要确保情境设计足够通用，让不同背景的人都能代入
