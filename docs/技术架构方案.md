# MindMatch 技术架构方案

> 版本: v1.0 | 日期: 2026-05-29 | 任务: Day 1 任务 1.1 产出物
> 核心原则: 轻量、模块化、零依赖、可独立调试

---

## 一、方案选择：为什么是纯 HTML + ES Modules

对比了 4 种候选方案：

| 方案 | 构建步骤 | 模块化 | 调试体验 | 部署复杂度 | 学习成本 | 结论 |
|------|---------|--------|---------|-----------|---------|------|
| React/Vue (Vite) | 需要 | 天然支持 | 需要 sourcemap | 需要 build | 高 | 太重 |
| React/Vue (CDN) | 无需 | JSX 受限 | 一般 | 简单 | 中 | 半吊子方案 |
| Svelte | 需要编译 | 天然支持 | compiler 层屏蔽 | 需要 build | 中 | 编译破坏"所见即所得"|
| **纯 HTML + ES Modules** | **无** | **天然文件隔离** | **F12 即调试** | **拖到 Vercel 即可** | **零** | ✅ 最佳 |

**项目计划书已确认纯 HTML/CSS/JS 方案。这里在此基础上增加 ES Modules 实现模块化。**

## 三、数据流

### 3.1 完整链路

```
[游戏页]                           [结果页]
    │                                  │
G1 完成 → store.save("game1", {...})   │
G2 完成 → store.save("game2", {...})   │
G3 完成 → store.save("game3", {...})   │
G4 完成 → store.save("game4", {...})   │
    │                                  │
    └──────── 跳转 results.html ────────┘
                                       │
                              store.loadAll()           ← 从浏览器存储读数据
                                       │
                              translator.generate(data) ← 纯函数：JSON → 文案
                                       │
                              insight.generate(data)    ← 调用混元 API (可选)
                                       │
                              matcher.match(data)       ← 规则引擎 + AI (可选)
                                       │
                              portrait.render(...)      ← 渲染 HTML
                              charts.render(...)        ← 渲染 ECharts
```

### 3.2 开发时 Mock 模式

每个模块**不需要依赖其他模块就能开发**：

```
# 开发游戏1时
test-game1.html → 只加载 game1-data.js + engine.js → 在浏览器看评分结果

# 开发翻译层时
test-translator.html → 只加载 translator.js → 从 mock/game-results.json 读取数据

# 开发洞察层时
test-insight.html → 只加载 insight.js → 从 mock/ai-responses.json 读 mock 响应
```

## 四、关键技术设计

### 4.1 数据总线 (store.js)

```
store.save(gameId, data)   → 写入浏览器存储
store.load(gameId)         → 读取单个游戏数据
store.loadAll()            → 读取全部四款游戏数据
store.clear()              → 清空（测试用）
store.hasAll()             → 检查四款游戏是否全部完成
```

- **生产环境**: 使用 `localStorage`
- **测试环境**: 使用内存对象（避免污染真实数据）
- **JSON Schema 校验**: 写入前校验 `数据结构定义.md` 中定义的数据格式

### 4.2 游戏引擎 (engine.js)

所有游戏复用同一套决策→评分管道：

```javascript
engine.run({
  scenarios: [...],        // 情境列表
  scoring: { ... },        // 评分规则（选项权重映射）
  onDecision: (data) => {},// 每个决策点的回调
  onComplete: (result) => {} // 全部完成后的回调
});
```

`engine.js` 不关心具体游戏内容（情境和权重在 gameX-data.js 中定义），只负责：
- 渲染决策界面
- 计时和悬停追踪
- 按权重累计维度分数
- 归一化输出 0-10（或 -1~+1）

### 4.3 API 调用 (api.js)

```javascript
api.callHunyuan(prompt, options)     // 调用混元 API
api.callWithFallback(prompt, options)// 带降级：混元失败 → DeepSeek
```

- **设计决策**: 前端直接调用混元 API（计划书确认，Demo 阶段 API Key 暴露可接受）
- **超时保护**: 10 秒超时，超时返回 null（触发降级逻辑）
- **错误处理**: 网络错误/API 错误 → 返回 null → 上层降级为纯规则模式

### 4.4 翻译层 (translator.js)

纯函数，零依赖，零异步：

```javascript
translator.generate(gameResults) → { sections: [...] }
```

每个维度按分数区间（0-3 / 4-6 / 7-10）匹配三档文案模板。不涉及任何 API 调用或 DOM 操作。

### 4.5 洞察层 (insight.js)

异步函数，依赖 api.js：

```javascript
insight.generate(translatedData) → { insights: [...] } | null
```

触发条件（在 prompt 中指定）：
- 同游戏内两维度差 ≥ 3 分
- 跨游戏矛盾或一致
- 某维度异常低（< 2 或 > 9）

失败（API 超时/报错/返回格式异常）→ 返回 null → 报告退化为纯翻译模式。

### 4.6 岗位匹配层 (matcher.js)

双轨并行：

```javascript
matcher.match(portraitData) → {
  ruleScores: { 沟通, 创意, 分析, 技术 },  // 规则引擎，始终有值
  aiInsight: "..." | null,                 // AI 补充视角，可空
  recommendations: [ { role, score, reason } ]
}
```

规则引擎：用 `数据结构定义.md` 第 6 节中定义的权重映射表加权计算。AI 仅生成补充推荐语。

## 五、模块独立性验证

| 模块 | 输入 | 输出 | 外部依赖 | 能否独立测试 |
|------|------|------|---------|------------|
| `gameX-data.js` | 无 | 情境/选项/权重 JSON | 无 | ✅ 纯数据文件 |
| `engine.js` | 情境+权重 JSON | 分数剖面 JSON | 无 | ✅ 可 mock 情境数据 |
| `store.js` | key+value | 无 | 浏览器存储 API | ✅ 可注入内存实现 |
| `translator.js` | 14维分数 JSON | 翻译文案 JSON | 无 | ✅ 纯函数 |
| `insight.js` | 翻译后 JSON | 洞察文案\|null | api.js | ✅ mock ai-responses.json |
| `matcher.js` | 肖像 JSON | 匹配结果 JSON | api.js(可选) | ✅ 规则引擎部分无依赖 |
| `api.js` | prompt | 文本\|null | 网络 | ✅ mock HTTP 响应 |
| `charts.js` | 数据 | DOM 更新 | ECharts CDN | ✅ 独立 HTML 测试页 |

**结论: 所有核心模块都能独立测试，游戏开发和结果页开发可以完全并行。**

## 六、开发工作流

### 6.1 启动本地开发

```bash
cd mindmatch-demo
python -m http.server 8080
# 浏览器打开 http://localhost:8080
```

### 6.2 开发游戏

```bash
open http://localhost:8080/test/test-game1.html   # 只测 G1
# 改代码 → 刷新浏览器 → 看控制台输出 → 调通后集成到 game1.html
```

### 6.3 开发结果页

```bash
open http://localhost:8080/test/test-translator.html  # 只测翻译
open http://localhost:8080/test/test-insight.html     # 只测洞察
open http://localhost:8080/test/test-matcher.html     # 只测匹配
# 三个都调通 → 集成到 results.html
```

### 6.4 首次部署

```bash
git add .
git commit -m "feat: complete Day 1 tech scaffold"
git push
# Vercel 自动检测 GitHub push → 自动部署 → 返回公网 URL
```
