## Day 0（2026-05-29）：项目启动——方案设计阶段

### 完成的工作
- 使用 /think 思维教练进行多轮方案讨论
- 确定核心定位：自我觉察工具（非招聘评估工具）
- 确定差异化叙事："帮你看到自己不曾察觉的地方"
- 通过 IMA 搜集心理学理论框架，整合为 4 个评估维度
- 读取腾讯官方四大方向岗位描述，生成 `四大方向特征提取.md`
- 确定游戏方向：投射式体验（情境投射 + 两难选择 + 角色代入）
- 确定技术栈：HTML/CSS/JS + Vercel + 混元 API + DeepSeek 备选
- 确定 AI 策略：规则评分 + AI 报告生成（方案 A）
- 生成完整项目计划书

### 关键决策记录

| 决策 | 选项 | 最终选择 | 理由 |
|------|------|---------|------|
| 产品定位 | 帮公司筛人 vs 帮应聘者理解自己 | 帮应聘者理解自己 | 差异化更强，思辨深度加分 |
| 游戏类型 | 行为测量型 vs 投射式自我觉察 | 投射式自我觉察 | 更符合"看到不曾察觉"的定位 |
| 游戏数量 | 3 个 vs 4 个 | 4 个（每个维度一个） | 结构更清晰，但容错空间为零 |
| AI 层级 | AI 评分 vs AI 报告生成 | AI 报告生成（规则评分） | 评分逻辑透明可解释；AI 擅长文案 |
| API 选型 | 混元 vs DeepSeek vs 其他 | 混元主 + DeepSeek 备 | 品牌对齐 + 免费额度 + fallback |
| 技术栈 | React/Vue vs 纯 HTML/CSS/JS | 纯 HTML/CSS/JS | 14天单人选不出错，部署简单 |

### 调整记录

| 原方案 | 调整后 | 原因 |
|--------|-------|------|
| 包含简历上传分析 | 砍掉 | 开发成本高，对评分权重低 |
| 覆盖所有功能 | MVP 思维 | 时间约束下的必要取舍 |
| 纯苏格拉底式引导（skill） | 增加理性决策评估 | 用户需要对决策进行审视 |

## Day 1（2026-05-29 晚）：方案定稿 + 数据结构定义

### 完成的工作
- SkillHub 技能搜索与安装（find-skills + drive-motivation + ux-writing）
- 创建 workflow-skill-matcher skill（项目→任务→技能自动匹配）
- 学术文献多轮搜索（McClelland, Schein, Riding, Steger, Gamified SJT, O*NET）
- 确定 4 个游戏的测量框架和输出指标（14 个连续分数维度）
- 定义完整数据结构（通用行为数据 + 各游戏输出格式）
- 确定 AI 肖像层两阶段策略（翻译基础 + 洞察加分）
- 产出 `数据结构定义.md`

### 关键决策记录

| 决策 | 选项 | 最终选择 | 理由 |
|------|------|---------|------|
| 数据粒度 | 单一标签 vs 连续剖面 | **连续剖面**（14维各独立计分） | 为 AI 肖像层提供深度数据 |
| G2 职业锚维度数 | 6 个 vs 8 个 | **8 个全部保留** | Demo 阶段先全量，后续可缩减 |
| G3 认知风格维度 | 2 维 vs 1 维 | **核心1维**（整体-分析），场独立可选 | 场独立与岗位匹配相关性较弱 |
| AI 肖像层策略 | 纯翻译 vs 纯洞察 vs **翻译+洞察** | **两阶段：翻译（必须）+ 洞察（尽力）** | 翻译保底线交付，洞察加分；AI 质量差时降级 |
| G2 评分逻辑 | 选什么加什么 vs **放弃什么才是锚点** | **放弃才扣分** | 锚点的本质是"不能放弃的东西" |
| Layer 3 岗位匹配 | 纯规则 vs 规则+AI | **规则主算分 + AI 辅助评估** | 规则兜底透明可解释；AI 补充视角；Day 8 细化 |

### 技术架构方案确定（23:54）

- 方案对比：React/Vue(太重) / Svelte(需编译) / CDN框架(半吊子) → **纯 HTML + ES Modules 胜出**
- 目录结构设计：games/ + js/core + js/games + js/layers + js/ui + mock/ + test/
- 数据总线：浏览器存储（游戏→结果页唯一桥梁）
- 游戏引擎：复用 `engine.js`（情境+权重在 gameX-data.js 中定义）
- Mock 驱动开发：每个模块可独立测试（test/test-*.html + mock/*.json）
- 外部依赖：仅 ECharts CDN（~1MB），核心代码零框架
- 产出 `技术架构方案.md`

### 匹配算法深度设计（00:28）

**用户要求**：
1. G1-G4 数据整合后统一输入匹配引擎
2. 需要合成数据集进行算法调参和测试
3. 肖像生成与岗位匹配独立看待

**产出物**：`匹配算法设计.md`

**核心设计**：

| 模块 | 方案 |
|------|------|
| 数据整合 | `integrator.js` — 从 store 读取G1-G4 → 14维标准化[0,1] → 统一profile |
| 合成数据集 | 4种岗位理想剖面 + 20个虚拟用户（清晰/混合/随机/极端×4类） |
| 匹配算法 | 双轨混合：轨道一（线性加权/规则兜底）+ 轨道二（TOPSIS距离法）+ 轨道三（AI辅助/加分） |
| 肖像与匹配关系 | 共享输入数据但完全独立运行，互不依赖 |

**学术参考（6篇）**：
- TOPSIS人事选拔对比研究 (MDPI Mathematics, 2024) → 核心算法
- PJFNN双塔匹配 (arXiv 1812.08947, 2018) → 架构概念
- 多维度特征融合PJF综述 (CSDN, 2023) → 融合思路
- LLM+Fuzzy TOPSIS (arXiv 2601.22433, 2026) → AI辅助模式
- Gallup人格岗位匹配 (MDPI Algorithms, 2025) → 映射验证
- Barrick & Mount元分析 (1991) → 理论基础

**代码规划**：
- `js/layers/integrator.js` — 数据整合
- `js/layers/matcher_linear.js` — 线性加权
- `js/layers/matcher_topsis.js` — TOPSIS
- `js/layers/matcher_ai.js` — AI辅助
- `js/layers/matcher.js` — 主编排器
- `mock/ideal-profiles.json` — 理想剖面
- `mock/synthetic-users.json` — 20个虚拟用户
